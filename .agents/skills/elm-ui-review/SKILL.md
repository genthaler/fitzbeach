# Elm UI Review Skill

## When to use
- Review an `elm-ui` interface for visual polish.
- Tighten a UI toward a calm, refined, minimal style.
- Check whether spacing, hierarchy, and interaction design feel too dense or too loud.

## When not to use
- Broad product redesigns unless explicitly requested.
- Logic-heavy Elm refactors where visual review is secondary.

## Workflow
1. Spacing and layout rhythm.
2. Hierarchy and typography.
3. Contrast and surface treatment.
4. Interaction clarity and focus states.
5. Final visual polish.

## Fixing guidance
- Prefer restraint over decoration.
- Reduce density before adding styling.
- Use subtle separation instead of heavy shadows or bright accents.
- Keep interactive controls obvious and accessible.
- Avoid dashboard-like visual treatment unless the project clearly calls for it.
- Do not redesign the interface unless the user asks for that level of change.
- For reviews, lead with concrete findings.
- For edits, make the smallest visual change that improves hierarchy or calmness.
- Point to the smallest worthwhile change rather than listing speculative improvements.
- Tie recommendations to the existing `elm-ui` structure instead of proposing a redesign.

## Final checks
- Findings or edits stay tied to the existing `elm-ui` structure.
- Reviews lead with concrete findings when issues are present.
- Any proposed visual changes remain calm, minimal, and accessible.
